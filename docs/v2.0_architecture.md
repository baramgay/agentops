## 1. Current Architecture Analysis (v1.0)

### 강점 — 유지할 것

| 영역 | 강점 |
|------|------|
| **단순성** | 단일 HTML + JSON 파일. 의존성 없음. 빠른 로딩 |
| **직관적 UX** | 탭 구조 명확. 에이전트 카드 클릭 → 상세 드로어 |
| **낮은 복잡도** | JavaScript 순수 바닐라. 프레임워크 없음 |
| **오프라인 우선** | fetch 실패 시 localStorage fallback |
| **다크 테마** | CSS 변수 기반. 일관된 디자인 시스템 |
| **실시간 폴링** | agent_status.json 5초 주기. 안정적 |
| **픽셀아트 사무실** | Phaser 3 기반. 에이전트 상태 시각화 독창적 |
| **수직 지휘체계** | OC→리드→담당 명문화. 검토 역순 구조 |

### 한계 — 개선 대상

| 영역 | 한계 | v2.0 해결 방향 |
|------|------|---------------|
| **observability** | 에이전트 상태 외 시스템 전체 상태 불가시 | System Overview 탭 |
| **토큰/비용 추적** | 없음 | Token Flow 패널 (demo 데이터) |
| **위험 감지** | 없음 | Risk 패널 |
| **조직 구조 시각화** | 텍스트 목록만 | Org 분포 차트 |
| **파이프라인 상태** | 다이어그램만, 실행 상태 없음 | Pipeline Status 패널 |
| **첫 화면** | 메타버스 사무실 (3D 렌더링 시간 소요) | Overview 탭이 첫 화면 |

## 2. v2.0 Target Architecture

### 핵심 원칙
- **공존 (coexistence)**: v1.0 파괴 없이 v2.0 레이어 추가
- **Feature Flag 우선**: 모든 v2.0 기능은 플래그로 제어
- **점진적 채택**: 모듈 단위 독립 활성화/비활성화
- **과설계 금지**: 실제 운영 가능한 수준만 구현

### 레이어 구조

```
┌─────────────────────────────────────────────┐
│           v2.0 System Overview Tab          │  ← NEW (feature flag)
│  [Status Bar] [Live Activity] [Token Flow]  │
│  [Org Chart] [Pipeline] [Risk] [Events]     │
├─────────────────────────────────────────────┤
│           v1.0 Core (보존)                   │
│  [Metaverse] [Agents] [Pipeline] [Editor]   │
│  agent_status.json polling (5s)             │
│  FastAPI :8765 + Static :8000               │
└─────────────────────────────────────────────┘

Feature Flags (V2_FEATURE_FLAGS):
  OVERVIEW_TAB      — 시스템 개요 탭 전체
  TOKEN_FLOW        — 토큰/비용 패널
  RISK_PANEL        — 위험 감지 패널
  GOAL_SYSTEM       — 목표 기반 운영 (Phase 3)
  GOVERNANCE        — 승인 워크플로 (Phase 3)
  AUDIT_TRAIL       — 감사 추적 (Phase 3)
```

### 데이터 흐름

```
agent_status.json  ──┐
AGENTS (inline JS)   ├──▶ overviewAdapter() ──▶ renderOverview()
SYSTEM_OVERVIEW_DATA ┘         │
    (mock/demo)                └──▶ V2_FEATURE_FLAGS 체크
```

## 3. Migration Strategy

### 단계별 이행

| Phase | 범위 | 방식 | 위험도 |
|-------|------|------|--------|
| Phase 1 (현재) | CSS + Feature Flags + Overview Tab HTML/JS | index.html 확장 | 낮음 |
| Phase 2 | 실제 API 어댑터 (token, audit trail) | adapter 계층 추가 | 낮음 |
| Phase 3 | Goal System, Governance, Approval | 별도 탭/모달 | 중간 |
| Phase 4 | 실시간 WebSocket 이벤트 스트림 | SSE or WS | 중간 |

### 불변 원칙
- 기존 6개 탭 구조 보존
- switchTab() 하위 호환 유지
- AGENTS 데이터 오브젝트 불변
- CSS 변수 시스템 불변
- agent_status.json 스키마 불변

## 4. Rollback Strategy

### 전체 롤백
```javascript
V2_FEATURE_FLAGS.OVERVIEW_TAB = false;
// → Overview 탭 숨김, 기존 메타버스가 첫 탭
```

### 모듈별 롤백
```javascript
V2_FEATURE_FLAGS.TOKEN_FLOW = false;   // 토큰 패널만 숨김
V2_FEATURE_FLAGS.RISK_PANEL = false;   // 위험 패널만 숨김
V2_FEATURE_FLAGS.GOAL_SYSTEM = false;  // 목표 시스템 비활성화
```

### localStorage 기반 플래그 지속성
```javascript
// 런타임에서 플래그 변경 가능
localStorage.setItem('v2_flags', JSON.stringify(V2_FEATURE_FLAGS));
```

## 9. Agent Interaction Model

```
사용자 요청
    │ (지시)
    ▼
orchestrator (OC)
    │ 목표 분석 + 팀 배정
    ▼
팀 리드 (lead-data / lead-dev / lead-pptx)
    │ 작업 분해 + 에이전트 배정
    ▼
담당 에이전트 (worker)
    │ 단위 작업 수행
    │ (검토)
    ▼
팀 리드 1차 검토
    │
    ▼
OC 최종 검토 → 사용자 전달
```

### 상태 전이
```
idle → working → done → (lead) review → (OC) review → idle
                      → waiting (블로커 발생)
                      → working (재작업 지시)
```
