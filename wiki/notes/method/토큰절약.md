# Claude Code 품질·효율 균형 전략 (Context Rot 방지)

## 핵심 원칙

AI 생산성은 **모델 성능 × 컨텍스트 관리 능력**으로 결정된다.
비싼 모델도 썩은 컨텍스트 위에서는 성능이 나오지 않는다.
반대로, 좋은 컨텍스트 + 적절한 모델 계층화 = Opus 4.8 수준 품질 + 합리적 비용.

---

# 현재 시스템 설정 (2026-06-09 기준)

```json
{
  "model": "opusplan",
  "effortLevel": "high",
  "MAX_THINKING_TOKENS": "50000"
}
```

## opusplan 이란?

Opus가 **계획·추론** (최대 50,000 thinking tokens), Sonnet이 **실제 출력** 생성.
→ Opus 수준의 사고 깊이 + Sonnet의 출력 효율 = 품질·비용 균형의 최적점.

## effortLevel = high 효과

Claude가 검증·탐색 단계를 생략하지 않는다.
- "이 코드가 깨질 경우는?" 자체 검토
- 대안 접근법 탐색 후 최선 선택
- 완료 선언 전 동작 확인

비용: 복잡한 작업에서 ~20-30% 토큰 증가. 단순 작업은 영향 없음.

## MAX_THINKING_TOKENS = 50,000

복잡한 문제에서 Opus가 더 깊이 추론한다.
단순 작업에서는 thinking이 트리거되지 않으므로 비용 무증가.

# 작업 유형별 모델 선택

| 작업 | 모델 | 전환 방법 |
|------|------|---------|
| 일반 코딩·분석 | opusplan (기본) | 자동 |
| 탐색·검색 (subagent) | Haiku | `model: "haiku"` 명시 |
| 보고서·문서 작성 | full Opus | `/model opus` |
| 아키텍처 설계·ADR | full Opus | `/model opus` |
| 30분+ 해결 안 된 버그 | full Opus | `/model opus` |
| 단순 반복 작업 | Sonnet | `/model sonnet` |

**승격 후 복귀 필수**: `/model opus` 사용 후 → `/model opusplan` 복귀

# 컨텍스트 관리 원칙

## /clear

### 사용 시점

완전히 다른 작업을 시작할 때

### 목적

- 컨텍스트 초기화
- 토큰 리셋
- 이전 작업 영향 제거

### 예시

```text
프로젝트 A 종료

/clear

프로젝트 B 시작
```

---

## /compact

### 사용 시점

작업 단계가 전환되는 시점

예:

```text
리서치 완료
↓
구현 시작
```

### 목적

- 핵심 정보만 유지
- 토큰 절약
- 컨텍스트 정리

### 주의

구현 중간에는 사용하지 않는다.

이유:

- 변수명 손실
- 함수명 손실
- 파일 경로 손실
- 세부 구현 맥락 손실

가능성이 있다.

# MCP 관리 원칙

MCP는 많을수록 좋은 것이 아니다.

모든 MCP는 컨텍스트 공간을 소비한다.

## 권장 기준

```text
MCP 서버 ≤ 10개
활성 도구 ≤ 80개
```

## 문제 상황

```text
MCP 서버 30개
도구 300개
```

↓

AI는 실제 작업보다

- 도구 탐색
- 도구 설명 이해

에 더 많은 토큰을 사용하게 된다.

# Continuous Learning

AI가 반복 패턴을 자동으로 학습한다.

## 학습 대상

- 코딩 스타일
- 프로젝트 구조
- 라이브러리 사용 패턴
- 네이밍 규칙
- 개발 습관

---

## 진화 과정

```text
반복 행동
↓
Instinct 생성
↓
신뢰도 증가
↓
Skill 승격
↓
재사용
```

결과적으로 같은 설명을 반복할 필요가 줄어든다.
